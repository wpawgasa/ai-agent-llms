# GRPO Investigation Report

## 1. What "re-generated and qualified by an LLM (Opus)" actually means in the pipeline

There are **two distinct mechanisms**, and it is important not to conflate them:

### A. The `dataset-verifier` agent — REPORT ONLY, never touches data
`/workspaces/ai-agent-llms/.claude/agents/dataset-verifier.md`

The agent is explicitly **read-only**. Its own frontmatter and body say so twice:
- L8: *"Read-only — never edits data."*
- L21-22: *"You are read-only. Never edit the dataset, the generator, or any source file. Your output is a report, not a code change."*

It runs two deterministic tools (`validate_dataset`, `quality_profiler`) then does a **qualitative LLM rationality read** of 5-8 sampled conversations (dialogue coherence, Thai naturalness, tool realism, behavior fidelity) and emits a prioritized verdict. It produces **no filtered/regenerated files**.

### B. The target-driven generation loop — this is what WRITES data
`/workspaces/ai-agent-llms/scripts/generate_sft_until_target.py`

This is the actual "re-generate + qualify" pipeline that changes DATA. Docstring (L4-22):
&gt; *"keeps generating, verifies each batch, removes unqualified samples, and loops until the qualified count reaches the per-level target — so the delivered corpus has a guaranteed number of clean samples."*

Its qualification bar (L13-17) is **purely deterministic**, using two library helpers, NOT the LLM:
- `clean_task_a_sft.clean_record` — drops truncated / role-confused rows, sets `terminal_reached`
- `quality_profiler.profile_task_a` + `defective_conversation_ids` — drops rows with **hard structural defects**

Critically, the LLM (`dataset-verifier`) is only an **advisory** step here. L24-26 and L44-46:
&gt; *"The *hybrid* final step — a semantic rationality verdict from the `dataset-verifier` LLM agent — is a Claude-driven post-step: this script writes a manifest and prints the exact follow-up command."*
&gt; *"--verify-batches … logged only — the deterministic filter still does the removal"*

**Answer to Q1:** The LLM ("Opus") qualification does **not** change the data — it only emits a report/verdict. Data removal/regeneration is done by the **deterministic** filters (`clean_record` + `profile_task_a`). The phrase "qualified by an LLM" refers to the advisory rationality verdict layered on top of a deterministic filter-and-regenerate loop.

### Criteria checked — all SURFACE quality, none reward-relevant

**`data_validator.py`** (`validate_dataset` / `_check_workflow_rationality`, L169-260) checks:
1. Required fields, graph shape (states/initial/terminal), first-message-is-system
2. Conversation-shape / strict-alternation (`find_shape_violations`)
3. Tool-state coherence — listed tools exist in `tool_schemas`; GT-called tools are listed in their state (L195-222)
4. Instruction completeness per non-terminal state
5. Terminal reachability via BFS (L241-258)
6. State-sequence continuity
7. **Thai-corruption** as a *warning only* — explicitly "does not affect `valid`" (L367-369): *"surfaces rows that should be dropped or regenerated before SFT."*

**`quality_profiler.py`** (`profile_task_a`) checks:
- **Hard structural defects** (must be 0): initial/terminal in states, edge endpoints in states, GT transition legality, message STATE-seq == `ground_truth.state_sequence`, role sequencing, tool_call JSON validity + schema membership (L203-265)
- **Spec-conformance** (advisory): `num_states` vs `target_path_len`, `num_tools` floor, `chain_depth`, back-edge/loop count (L180-201)
- **Structural-ceiling** analysis: undershoot vs hard impossibility (L275-306)
- **Distributions**: behavior mix, language, self-loop share, arrow glyphs, tool-chain propagation, recovery coverage

### Does any of it touch reward-relevant properties? — NO

Neither validator scores or gates on **within-group variance**, **tool-call presence relative to a GRPO group**, or **state-transition diversity across rollouts**. Note the profiler *computes* diagnostics that look adjacent — `_propagation_hops` (L106-129, tool-chain propagation) and `self_loops` share — but these are **descriptive distribution stats on the static ground-truth corpus**, not reward-signal properties. Tool-call presence is only checked for schema membership / placement legality (does the GT tool call belong in its state), never for *whether a prompt will elicit varied tool emission across policy rollouts*.

The GRPO diagnosis doc confirms this gap directly (`grpo_diagnosis_gemma4_26b.md` L1145-1147):
&gt; *"`frac_reward_zero_std ≈ 77%` on the new data — worse than `df4dot2d`… LLM-qualifying the corpus did not, on its own, produce the within-group reward spread GRPO needs."*

**So qualification/regeneration touches only surface quality (schema, dedup, rationality, structural conformance) — it is blind to every reward-relevant property.**

## 4. Candidate structural fixes the doc already enumerated

From `/workspaces/ai-agent-llms/docs/grpo_diagnosis_gemma4_26b.md`, the "Three Fix Candidates" section (L279-377) plus the later reframing. The findings that motivate them:
- **Reward-lattice / bucket-clumping** (L716-724): active components pool into *"a small finite reward lattice… at most ~16 possible reward values per prompt"* — 8 diverse completions land in 1-3 buckets.
- **Tool-emission gap** (L775-795): *"In all 10 collapsed groups: 0/8 rollouts attempt a tool call"* — the model deterministically tool-skips.
- **Saturated-correct** (L782, L956 step 49): 6/10 collapsed groups are *"8 different ways of saying the same correct thing"* scored identically — *"fundamentally unreachable by any tool- or state-side reward change."*

### The candidate fixes, with the doc's stated pros/cons:

**(a) Row-per-conversation / single-turn target** (L281-315)
- **Pro:** Fixes train/deploy distribution mismatch; raises effective `task_completion` weight from ~1.4% → ~8.4% (fires on 83.5% of rows instead of 14.3%).
- **Con:** Does **not** fix within-group variance — *"same prompt + same policy = same completions"* (L307-309); `chain_propagation` stays dead (single turn, `n_pairs ≤ 1`); 5.86× smaller dataset → ~6× fewer optimizer steps (L313-314). Explicitly **re-classified as a reward-calibration/alignment fix, NOT the variance lever** (L286-289).
- **Code scope:** ~30 lines in `_load_grpo_jsonl`.

**(b) Drop saturated components / reweight** (L338-355) — *landed*
- **Pro:** Zero-risk cleanup; reweights binary state/tool from 0.30→0.40 each for slightly stronger gradients.
- **Con:** *"Effect on reward variance: marginal"* — the dropped components contributed 0 variance anyway.
- **Code scope:** ~5 lines in `reward_business_logic.py`.

**(c) Graded state/tool components (continuous scoring)** (L357-377) — *landed*
- **Pro:** *"highest leverage"* — per-argument tool F1 (0.4 name + 0.6 pro-rata args) and graded state matching spread completions across a range; *"Expected mean_reward_std jumps from ~0.05 to ~0.15+"* (L372-373).
- **Con:** Risk of divergence between training reward and benchmark eval — must keep the strict metric separate (L374-376). And it *"can't manufacture variance"* where the policy emits identical/saturated-correct completions (L456-461).
- **Code scope:** ~150 lines + tests; ~half day.

**(a\*) Multi-turn rollouts with simulator** (L316-335, L1003-1005) — *deferred, the "real" lever*
- **Pro:** Makes `chain_propagation` and `task_completion` per-trajectory signals that vary across K rollouts; *"Within-group reward_std should jump substantially: the variance space is N-fold larger (one decision per turn × N turns)"*; best deployment alignment.
- **Con:** *"~1 week of engineering, plus meaningfully more rollout compute"*; simulator bugs *"that silently corrupt training signal"* need unit tests + held-out replay validation (L333-335); crucially **does not help the "saturated correct" case** (L1005).

**A continuous prose-quality component** was later added as a candidate specifically for the saturated-correct groups (L792-793, L811): *"semantic similarity to GT assistant message via sentence-transformers"* — the only reward-side lever that could differentiate 8 correct prose surfaces.

**Doc's recommended ordering** (L378-404): land **(b) first** (free cleanup) → **(c)** (variance where signal exists) → retry GRPO with hygiene fixes (ckpt-500, `beta=0.1`, `loss_type=grpo`, `max_completion_length=512`) → **(a)** for alignment only → **(a\*)** as last resort if (b)+(c) don't lift `frac_reward_zero_std`.

# GRPO reward-variance root-cause: findings

## 1. `reward_business_logic.py` — reward structure &amp; discretization

**Component weights** (`reward_business_logic.py:41-44`):
- `W_STATE_TRANSITION = 0.40`, `W_TOOL_CALL_F1 = 0.40`, `W_TASK_COMPLETION = 0.10`, `W_TRANSITION_LEGALITY = 0.10`.

**`_graded_state_match` tiers** (`reward_business_logic.py:82-113`):
- Empty GT: `1.0 if not predicted else 0.0` (`:100-101`).
- Per `(pred, gt)` pair: exact `(from,to)` match → `+1.0` (`:107-108`); `from`-only **or** `to`-only match → `+0.5` (`:109-110`); reverse direction `pred[0]==gt[1] and pred[1]==gt[0]` → `+0.3` (`:111-112`); no overlap → `+0.0`. Score = `total / len(ground_truth)` (`:113`).
- **On a single-turn row `len(ground_truth)==1`** (the loader emits `state_seq = [state_trans]`, one element — `grpo.py:264-265`). The loop only compares index `i=0`, i.e. `predicted[0]` vs the single GT pair. So `r_state` collapses to exactly one of **{0.0, 0.3, 0.5, 1.0}** — 4 discrete rungs, and only the model's *first* emitted transition can move it.

**`graded_tool_call_f1` on no-tool rows (GT empty)**: delegates to `compute_argument_graded_f1` (`reward_utils.py:50-60`), whose head is `if not ground_truth and not predicted: return 1.0` / `if not ground_truth or not predicted: return 0.0` (`tool_call_f1.py:277-280`). So on a no-tool GT row it is **binary {1.0, 0.0}**: 1.0 iff the model emits no parseable tool call, 0.0 the instant it emits any.

**Active-component mean + renormalization** (`reward_business_logic.py:168-192`): components start as `[(0.40, r_state), (0.40, r_tool)]`. `transition_legality` `(0.10, r_legality)` is appended only if `valid_transitions` is non-empty (`:173-176`); `task_completion` `(0.10, r_completion)` only if `terminal_reached` (`:181-184`). Then `total_weight = sum(w)` and `score = sum(w*s)/total_weight` (`:186-191`), renormalized so dropping components keeps `score ∈ [0,1]`. **Clamp**: `max(0.0, min(1.0, score))` (`:192`).

**Where the reward is forced onto a small discrete set (single-turn rows):**
- Per the loader, non-final rows have `terminal_reached=False` (`grpo.py:268-269, 281`), so `task_completion` is dropped; `valid_transitions` come from the workflow graph and are present, so legality stays active. Active set = state(0.40) + tool(0.40) + legality(0.10), `total_weight = 0.90`.
- `r_state ∈ {0, 0.3, 0.5, 1.0}` (4 values).
- `r_tool` on no-tool rows ∈ {0, 1}; on 1-GT-tool rows it collapses too (see §2).
- `r_legality = transition_legality_score` = fraction of emitted transitions that are legal edges or self-loops, `0.0` if none emitted (`reward_utils.py:112-135`); with a single emitted transition it is **{0, 1}**.
- `r_completion ∈ {0, 1}` (`reward_business_logic.py:183`).

So a typical single-turn, no-tool, non-terminal row yields `score = (0.40·r_state + 0.40·r_tool + 0.10·r_legality)/0.90` with `r_state∈{0,0.3,0.5,1}`, `r_tool∈{0,1}`, `r_legality∈{0,1}` → a **small finite lattice of possible reward values** (at most ~16 combinations, fewer in practice). This is the core variance-starvation fact.

## 2. `compute_argument_graded_f1` / `_pair_match_graded` — the cliffs

**`_pair_match_graded`** (`tool_call_f1.py:227-255`), `name_weight=0.4`:
- Name mismatch → `0.0` (`:239-240`) — no partial credit for arg overlap.
- Non-dict args → `1.0 if _deep_equals else name_weight(0.4)` (`:244-245`).
- Names match, GT has no args → `1.0` (`:247-248`).
- Names match, GT has args → `name_weight + (1-name_weight)·(matched/len(gt_args)) = 0.4 + 0.6·arg_frac` (`:250-255`).

**`compute_argument_graded_f1`** (`tool_call_f1.py:258-309`): both empty → `1.0` (`:277`); exactly one empty → `0.0` (`:279`); else greedy best-first 1:1 assignment (`:288-302`), `partial_tp = Σ pair_scores`, `precision = partial_tp/|pred|`, `recall = partial_tp/|gt|`, `F1 = 2pr/(p+r)`, with `precision+recall==0 → 0.0` (`:307-308`).

**Collapses to exactly 0.0**: GT non-empty and pred empty (or vice-versa); or every predicted name mismatches every GT name (all pair scores 0 → no pairs → `partial_tp=0`).
**Collapses to exactly 1.0**: both empty; or `|pred|=|gt|=1`, name matches, GT has no args (or args stripped to `{}` by `_strip_placeholder_args`, `reward_business_logic.py:116-131`) → pair `1.0`, `p=r=1`.
**Single-turn 1-GT-tool geometry**: with `|pred|=|gt|=1`, `F1` equals the single pair score, so `r_tool ∈ {0.0} ∪ {0.4 + 0.6·(m/K)}` for `K` GT args and `m` matched → e.g. K=1 gives **{0, 0.4, 1.0}**, K=2 gives **{0, 0.4, 0.7, 1.0}**. Emitting an extra (2nd) tool call rescales F1 by ~2/3 but stays discrete. Net: on single-turn rows the "graded" tool reward is still only a handful of values.

## 3. `compute_transition_accuracy` / `parse_state_transitions` (strict, held-out path)

`parse_state_transitions` (`state_accuracy.py:64-79`) regex-extracts `(from,to)` tuples from assistant messages; a completion is wrapped as one assistant message by `extract_state_annotations` (`reward_utils.py:17-22`).

`compute_transition_accuracy` (`state_accuracy.py:103-126`): empty GT → `(1.0 if not predicted else 0.0, 0)` (`:112-113`); else `correct = #{i : i&lt;len(gt) and pred[i]==gt[i]}`, `accuracy = correct/len(ground_truth)` (`:119-125`). **For a single-turn row `len(gt)==1`, only `i=0` is checked, so `accuracy ∈ {0.0, 1.0}` — strictly binary.** This strict scorer is what `state_sequence_match` wraps (`reward_utils.py:32-40`) and is used **only** in the held-out composite (`grpo.py:437`, `_heldout_composite_score`), deliberately kept numerically independent of the graded training reward (`grpo.py:398-411`). The training reward instead uses the graded `_graded_state_match` (§1).

## 4. `grpo.py` — reward adapter feed + generation geometry

**Adapter → TRL**: `reward_funcs=_make_reward_adapter(reward_fn)` (`grpo.py:818`). The adapter is keyword-only (`grpo.py:334-340`) to match TRL 0.23.1, JSON-decodes the `ground_truth` column, aliases `state_sequence → state_annotations` reshaping dicts to `(from,to)` tuples (`:350-361`), flattens chat completions to assistant strings (`:363-370`), and returns `reward_fn(prompts, flat_completions, gts)` (`:389`). Single reward function ⇒ TRL `reward_weights=[1.0]`, default `multi_objective_aggregation="sum_then_normalize"`.

**Geometry** (TRL divisibility in `grpo_config.py:882-919`; `steps_per_generation = generation_batch_size/(per_device_train_batch_size·num_processes)`; advantages are normalized per group of `num_generations` at generation time over the whole generation batch, then consumed across `steps_per_generation` optimizer steps):

- **Diagnostic** (`grpo_cat_a_diagnostic.yaml`: `num_generations=16`, `generation_batch_size=32`, `per_device_train_batch_size=8`, `gradient_accumulation_steps=1`; single GPU): `32 % 16 = 0 → 2 prompt-groups per generation batch` = **2 unique prompts per generation round**, each with **16 samples** for the within-group std estimate. `steps_per_generation = 32/8 = 4`.
- **Production** (`grpo_cat_a.yaml`: `num_generations=8`, no `generation_batch_size`, `per_device_train_batch_size`/`gradient_accumulation_steps` unset → TRL defaults 8/1): `generation_batch_size = 8·1·1 = 8`, `8 % 8 = 0 → 1 prompt-group` = **1 unique prompt per optimizer step**, 8 samples. This is the "df4dot2d ran 8/8 = 1 prompt/step → high prompt-draw noise" case (`grpo.py:648`).

So the diagnostic doubles prompt diversity (2 vs 1) and doubles group size (16 vs 8), i.e. more samples over which the discrete reward can vary.

## 5. TRL 0.23.1 advantage standardization &amp; loss_type

Installed at `.venv/lib/python3.12/site-packages/trl/trainer/grpo_trainer.py`.

**Advantage computation** (`sum_then_normalize` branch, `grpo_trainer.py:1964-1990`):
- `rewards = (rewards_per_func · reward_weights).nansum(1)` (`:1966`).
- `mean_grouped_rewards` = per-group mean over `num_generations`, `repeat_interleave`d (`:1967-1968`).
- `if scale_rewards in ["group","none"]: std_rewards =` per-group `std` over `num_generations`, repeat-interleaved (`:1971-1973`); for `"none"` this std is computed **only for zero-std logging** (`:1970`).
- `advantages = rewards - mean_grouped_rewards` (`:1987`) — always mean-subtract.
- `if scale_rewards != "none": advantages = advantages / (std_rewards + 1e-4)` (`:1988-1989`).

**`scale_rewards="group"` (TRL default, `grpo_config.py:872,697-698`) vs `"none"`**: `"group"` divides each advantage by its per-group reward std (+1e-4). On the near-constant discrete groups from §1–§3, that std is ~0.003–0.04, so the `+1e-4` floor barely helps and advantages are amplified 60–1060× — the documented instability (grad-norm 1126, KL 40) in `grpo.py:642-644`. `"none"` skips the division entirely (`:1988`): advantages are the raw group-mean-centered rewards, magnitude ≈ the reward spread itself. This is the "primary fix" (`grpo_cat_a_diagnostic.yaml:56`).

**loss_type handling** (`grpo_trainer.py:2388-2409`):
- `"grpo"`: `loss = ((per_token_loss·mask).sum(-1) / mask.sum(-1).clamp(min=1)).mean()` (`:2389`) — **per-sequence** normalization by each completion's own length → the documented short-completion length bias.
- `"bnpo"`: `(per_token_loss·mask).sum() / mask.sum().clamp(min=1)` (`:2393`) — normalize by total active tokens in the **local** batch (batch-size-sensitive).
- `"dr_grpo"`: `(per_token_loss·mask).sum() / (per_token_loss.size(0) · self.max_completion_length)` (`:2397`) — normalize by a **fixed global constant** (`batch × max_completion_length`), independent of actual completion length → removes length bias.
- Default `"dapo"` (`grpo_config.py:709`): normalize by `num_items_in_batch/num_processes` (`:2400-2402`).

**What `dr_grpo` changes about length normalization**: it replaces the per-sequence `1/len` denominator (which up-weights short completions with positive advantage / long with negative — the mechanism behind the `211→29`-token collapse noted at `grpo.py:635-636`) with a constant `1/max_completion_length` denominator applied uniformly, so gradient magnitude no longer depends on completion length.

## Bottom line
On single-turn rows the per-prompt reward is confined to a small discrete lattice: strict state accuracy is binary {0,1} (§3); graded state is {0,0.3,0.5,1} (§1); tool F1 is binary on no-tool rows and ~3–4 valued on 1-tool rows (§2); legality and completion are binary. The active-mean over `{0.40, 0.40, 0.10}` weights (`total_weight=0.90` on non-terminal rows) maps those onto a finite set of composite reward values. GRPO subtracts the group mean and (default `scale_rewards="group"`) divides by that group's tiny std (§5), so whenever the whole group of 8/16 lands on the same rung the advantage is 0 (`frac_reward_zero_std≈1`), and when it barely splits the `÷std` explodes the advantage. The diagnostic config attacks exactly this: `scale_rewards="none"` (no ÷std), `loss_type="dr_grpo"` (no length bias), and 2 prompts × 16 samples/group instead of 1 × 8 to give the discrete reward more chances to vary within a group.</result>

# DECISION MEMO — Go/No-Go: Online GRPO for Gemma-4-26B-A4B Cat A
**Date:** 2026-07-07 · **Runs killed to date:** 4 · **Author:** ML strategy review

---

## 1. VERDICT

**No — single-turn online GRPO with the current reward cannot learn a useful policy on this data as configured, and the just-merged stabilization does not change that. Conditional revival only if a directly measurable bar is met:**

&gt; Online GRPO is admissible only if a preflight on the actual GRPO training distribution (T=0.8, 16 samples/prompt, SFT ckpt-1000) measures **`frac_collapsed_groups` &lt; 0.50 AND median within-group `reward_std` ≥ 0.05**. Every measurement to date fails this bar: 116/150 steps (~77%) at whole-batch zero reward std on `bqbxnqxw` (`checkpoints/grpo_cat_a/.../checkpoint-150/trainer_state.json`), median 8 reachable reward values per prompt, 22% of prompts with ≤2 reachable values, and the 2026-05-26 preflight's "diverse text, identical scores" finding (7/8 unique completions, near-zero reward spread).

The mechanistic argument, from the advantage math rather than the run history:

- GRPO's gradient per group is ∝ (r_i − r̄). On a lattice where the whole 16-sample group lands on one rung, the gradient is **exactly zero** — no hyperparameter recovers it. With `scale_rewards="none"` (the fix), a 2-rung group with spread 0.02 produces advantages of ±0.01: nonzero but ~50× weaker than a healthy continuous-reward group. So the stabilized config converts "explode on 77% of steps" into "no-op on ~77% of steps and whisper on most of the rest." That is fact D's predicted STABLE-BUT-FLAT, and I concur with the prediction.
- **Step economics:** diagnostic geometry is 2 unique prompts/step. A 1000-step production run = 2,000 prompt-groups = 32,000 generations, of which (at 77% dead) only ~460 groups carry any gradient — ~3% coverage of the 14,682-row corpus, on coarse rungs, at lr 1e-6. The expected held-out composite movement is below the noise floor of the eval (200-prompt held-out, single seed).
- **The premise mismatch is structural, not a bug:** the reward was designed for trajectories, but the data feeds it single turns. `chain_propagation` is dead-constant (no `state_sequence` ≥ 2 anywhere), `task_completion` is dropped on 85.8% of rows, tool reward is binary on the 61.1% zero-tool rows. Three of five reward components are inert or binary on the majority of the data **by construction of the row layout**. No reward-side reweighting fixes a support problem.
- **Saturated-correct groups are not a problem to be solved** — a group where 8 phrasings all score 1.0 is the reward correctly reporting "nothing left to learn here." Manufacturing variance inside them (lever c) optimizes an off-target axis. The correct treatment is to *exclude* them from training, which online GRPO cannot do cheaply and offline methods do for free.

## 3. THE RECOMMENDATION

**Abandon single-turn online GRPO. Adopt Path #1 — rejection-sampling fine-tuning (RFT) using the existing graded reward as an offline ranker — gated by a 1-day headroom probe, with #2 (DPO/KTO) as a second stage built from the same sample corpus. Keep the graded reward; retire the online optimizer.**

Why #1 beats the runner-up (#3, running the already-built stabilized GRPO):

1. **Identical compute, strictly greater extraction.** A 1000-step GRPO run and a full RFT sampling pass generate the same order of completions (~32k vs ~117k for full corpus, ~same for matched budgets). GRPO consumes them online, paying full price for the ~77% dead groups and extracting a coarse-lattice gradient from the rest at lr 1e-6. RFT consumes the same completions offline: every prompt with any reward spread contributes a full-strength SFT target, and dead/saturated prompts cost only their sampling tokens. There is no configuration of GRPO on this reward that extracts more information per generated token than argmax-and-distill does.
2. **It removes the entire failure class that killed four runs.** All four post-mortems trace to the advantage-normalization machinery interacting with near-zero group variance (÷std amplification → grad 1,126 then 50,853; KL 40 then 529). RFT has no advantage, no group std, no KL controller, no reference policy — the discrete lattice, which is *fatal* to GRPO's variance requirement, is *sufficient* for RFT's ranking requirement. Fact A's lattice is exactly enough structure to rank and no more; use it for ranking.
3. **The stabilized diagnostic answers a question whose answer doesn't change the plan.** Fact D already predicts STABLE-BUT-FLAT; confirming it costs 2–4 H100-hours and, per this project's demonstrated pattern (10 addenda), invites another iteration cycle on a dead mechanism. The 12-minute preflight measures the same decision-relevant quantities (`frac_collapsed_groups`, `reward_std`) plus the RFT headroom that GRPO telemetry can't measure. Run the GRPO diagnostic only in the branch where the preflight *surprises us* (see §5).
4. **Honest ceiling, honestly priced.** RFT cannot exceed best-of-8 of the current policy. But with beta=0.05 KL-anchoring and lr 1e-6, neither could this GRPO — it was never going to explore meaningfully beyond the SFT policy either. If best-of-8 headroom is real (probe will tell us in a day), RFT captures it at near-zero risk; if it's not, *no* single-turn method works and the answer is #5 (multi-turn env) or shipping SFT-only — a decision worth reaching in 1 day instead of after a fifth kill.

Explicitly rejected: lever (c) semantic-similarity component (predicted reward-hack, precedent on file), and #5 multi-turn env as the *next* step (right ceiling, wrong sequencing — 1 week of net-new simulator is not justified before a 1-day probe of the cheap path).

## 4. NEXT-EXPERIMENT SPEC — "Headroom Probe → RFT Pilot" (gated, &lt;1 H100-day total)

**Base checkpoint for all stages:** `checkpoints/sft_cat_a/gemma-4-26B-A4B-it/checkpoint-1000` (loss-elbow seed per the 2026-07-07 addendum; ckpt-500 as fallback). First resolve the data-provenance caveat: verify `data/output/grpo/task_a/{train,validation}.jsonl` hashes against the W&amp;B `uklfswk5`/`bqbxnqxw` artifacts (validation.jsonl lags train.jsonl by 8 days).

### Stage 1 — RFT pilot (~6–10 H100-hours, only on Stage-0 GO)
- **Sampling:** N=8, T=0.8 over a 4,000-prompt stratified subset (bias toward tool-bearing and terminal rows). ## 5. WHAT WOULD CHANGE THIS

1. **Preflight surprises high:** `frac_collapsed_groups &lt; 0.50` with median `reward_std ≥ 0.05` at the diagnostic's exact sampling settings → the §1 condition is met, GRPO's death certificate is withdrawn, and the stabilized 50-step diagnostic (already merged, zero eng cost) becomes the correct next run — online RL then plausibly beats RFT by also learning from negatives.
2. **No headroom anywhere:** `frontier_frac &lt; 10%` AND greedy composite already ≥ target → the SFT policy is effectively at the single-turn ceiling; the correct move is to *stop training on this row layout entirely* — ship SFT-only for Phase 2 Cat A, or fund the multi-turn env (#5) if the E2E Phase 4 numbers demand more.

## 1. `_replay_conversation` — the eval-side conversation-replay skeleton

**File:** `src/llm_workflow_agents/eval/agent_benchmark.py:433-597`

**Signature** (`agent_benchmark.py:433-440`):
```

                     enable_thinking=False, engine="vllm")

```

**What it does:** Walks the ground-truth `sample["messages"]` list turn-by-turn. It rebuilds a sliding `context` and, at each **assistant** turn, calls the model (`_call_vllm`, `agent_benchmark.py:499`) instead of using the GT assistant text; system/user/tool turns are passed through from GT. Returns the model-substituted message list plus per-turn latency/TTFT.

Key behaviors relevant to multi-turn:
- **Tool results are NOT executed — they are replayed from ground truth** (`agent_benchmark.py:582-595`, comment "Use ground-truth tool response to avoid cascading failures"). It only re-threads `tool_call_id` from the model's actual tool call (`:590-593`).
- **User turns are NOT simulated** — they come verbatim from `sample["messages"]` (`:466-468`). There is no next-user generation.
- Handles a system→assistant preamble edge case with no model call (`:478-497`), and synthesizes structured `tool_calls` from inline `&lt;tool_call&gt;` text (`:517-534`).
- Does NOT early-exit on terminal state; walks every GT turn (`:567-580`).

**Training-usable?** **Eval-only as written.** It is a module-level function reachable only from the `__main__` benchmark block (callers at `agent_benchmark.py:746` and `:800`). It talks to an **HTTP OpenAI-compatible endpoint** (`_call_vllm`, `agent_benchmark.py:273-430`, uses streaming `chat.completions`), not an in-process `model.generate`. So its rollout loop is reusable as a *pattern* for multi-turn trajectory generation, but it is coupled to a served endpoint and to GT tool/user turns — it is not wired into the trainer, and it produces no rewards/advantages. Net: the replay *skeleton* exists; the training integration does not.

## 2. Current GRPO rollout — single-turn-per-row

**`_load_grpo_jsonl`** — `src/llm_workflow_agents/training/grpo.py:161-302`. This is the flattening step. ## 2. Per-conversation gold trajectory (the spine + tool chain)

Yes — a single conversation sample fully recovers both the intended state path and the intended tool-call sequence:

- **Full gold state path**: `ground_truth.state_sequence` is the complete ordered list of `{from,to}` transitions the reference trajectory traverses (11 entries in the sample, ending at the terminal). This is the spine-plus-branches actually walked, not just the canonical spine. You can reconstruct the node visitation order as `[state_sequence[0].from, then each .to]`.
- **Full gold tool chain, two granularities**:
  - `ground_truth.tool_calls` — flat ordered `list[{name, arguments}]` (5 in the sample), the entire chain.
  - `ground_truth.tool_chain_dependencies` — same calls grouped per assistant turn (`list[list[...]]`), preserving turn boundaries so you can align tool calls to state transitions.
- **Terminal check**: `ground_truth.terminal_state` (str) and `ground_truth.terminal_reached` (bool) let you score "did the rollout end in the correct terminal."
- Per-turn alignment is also available inline: each assistant message's `annotations.state_transition` and `annotations.tool_calls` (e.g. `{"state_transition":{"from":"ALERT_RECEIVED","to":"ASSESS_SEVERITY"},"tool_calls":[]}`), so you can map each gold transition to the exact turn that produced it.

So scoring "did the multi-turn rollout traverse the gold `state_sequence` and emit the gold tool chain" is fully supported by one sample. Construction logic: `_extract_ground_truth`, `generate_workflows.py:667-741`.

## 3. Workflow graph as a deterministic environment — with a serialization caveat

Edge structure in the SERIALIZED graph (`workflow_graph.transitions`) is exactly `{from, to, condition, priority}` (`generate_workflows.py:230-238`). Measured across all 290 validation graphs:

- **`priority`**: only two values — `0` (spine edges, n=4668) and `1` (branch/loop/optional edges, n=3122).
- **`condition`**: opaque synthetic labels only — `proceed_from_&lt;state&gt;` on every priority-0 spine edge, and `branch_S#_to_S#` on every priority-1 branch edge. There are 1866 branch points (states with &gt;1 outgoing edge) across the 290 graphs.

What a deterministic environment CAN do from this:
- **(a) Validate a proposed transition is legal**: yes. `(from,to)` adjacency is fully specified. This is already how the reward works — `grpo.py:239-243` builds `valid_transitions = [[from,to], ...]` from `workflow_graph.transitions` and passes it as `valid_transitions` into the row ground truth (`grpo.py:282`) for a transition-legality reward component.
- **(b) Decide the next state given the model's action**: only partially/heuristically. The serialized `condition` is a human-readable label, NOT an evaluable predicate, and `priority` is just spine(0) vs branch(1). The machine-evaluable trigger semantics — `_VALID_TRIGGERS_SET = {always, tool_success, tool_error, intent_match, slot_present, user_declines}` (`generate_workflows.py:476-479`) — live on the in-memory `WorkflowTransition.trigger`/`.optional` fields (`:203-204`) and are **NOT emitted by `to_dict()`**, so they are absent from the JSONL. The reference traversal `walk_path` (`generate_workflows.py:482-545`) resolves branches by simulating triggers stochastically (`rng.random() &lt; TOOL_ERROR_RATE`, behavior/intent-conditioned coin flips at `:519-530`) and then ranking edges by `edge_priority` (`:533-540`). So next-state selection at branch points was non-deterministic at generation time and its deciding inputs (trigger type, optional flag) are not in the data.

Net: from the serialized graph you get a deterministic legality validator and a spine-vs-branch preference (via priority), but not a deterministic condition-evaluating transition function. Note also that the prose `workflow_script` field (a rendered string, e.g. `- On success: proceed to [ASSESS_SEVERITY]` / `- If ...`) encodes some trigger semantics in natural language, but not machine-parseable structured form.

## 4. `_load_grpo_jsonl` — conversation → per-turn rows

`grpo.py:161-302`. It does NOT store per-turn rows on disk; `train.jsonl`/`validation.jsonl` are per-conversation (same schema as §1), sliced into per-turn rows at load time.

- **Slicing rule** (`grpo.py:245-252`): find all assistant message indices; keep only those whose immediately preceding message role is `user` or `system` (`valid_pairs`). Assistant turns preceded by a `tool` message are **skipped** (TRL rejects `tool` as the last prompt role) — this drops tool-response continuations. Docstring: `:161-191`.
- **Prompt per row** (`grpo.py:255-261`): `raw_msgs[:asst_idx]` — i.e. the FULL prior context (system + all prior user/assistant/tool turns) up to but excluding the target assistant turn, each slimmed to `{role, content}` via `_slim_content`. The leading system message is re-enriched via `build_enriched_system_prompt(raw, ...)` (`:216-224`) so rollouts see the same system prompt as the benchmark. So the prompt is NOT just system+one-user; it grows with each successive turn (prefix of the conversation).
- **Target / reference per row** (`grpo.py:271-283`): a `ground_truth` JSON-string column containing:
  - `state_sequence` = the SINGLE `{from,to}` from this turn's `annotations.state_transition` (`:264-265`) — per-turn, not the whole spine.
  - `tool_calls` = this turn's `annotations.tool_calls` only (`:266`).
  - `messages` = just this one assistant message's slimmed content (`:274-279`).
  - `terminal_state` = conversation-level value; `terminal_reached` = True ONLY on the final emitted row of a conversation whose overall `terminal_reached` was true (`:268-270, 281`).
  - `valid_transitions` = the full conversation's legal `(from,to)` edge list (`:282`).
  - There is no free-form "target completion" text stored; the reference for reward is this structured `ground_truth`, decoded in `_make_reward_adapter` (`grpo.py:326-361`).
- **Rows per conversation**: one per `user|system → assistant` boundary. Measured on the 290-conversation validation split: **1609 rows total, avg ≈ 5.55 per conversation, min 1, max 28** (assistant turns that follow `tool` messages are excluded from the count).

### Key implication for a trajectory-level reward
The existing GRPO pipeline is **per-turn / single-step**, not trajectory-level: each row's reference is one transition + one turn's tool calls, and `chain_propagation`/completion sub-rewards are explicitly neutralized for single-turn rows (`grpo.py:183-191`). The conversation-level gold trajectory you need (`ground_truth.state_sequence` full path, `ground_truth.tool_chain_dependencies`, `ground_truth.terminal_state/terminal_reached`) exists intact in the on-disk per-conversation JSONL (§1-2) but is currently discarded by the `_load_grpo_jsonl` slicing — a trajectory reward would need a different loader that keeps the whole conversation's ground_truth per row.</result>

## 1. Objective &amp; acceptance criteria

### Objective
Replace the per-turn single-completion GRPO rollout with a **whole-conversation replay rollout**: the model free-runs its own assistant turns against the gold conversation's scripted user turns and tool results, truncating when it leaves the gold path, and receives **one scalar trajectory reward** = 0.40·coverage + 0.40·graded tool-F1 + 0.10·terminal + 0.10·legality. Aggregating over a median-13-turn gold path turns the single-turn discrete lattice ({0, 0.3, 0.5, 1.0} per turn) into a ~near-continuous distribution (coverage alone has T+1 rungs, T = gold length), manufacturing the within-group reward variance GRPO's advantage needs.

### Acceptance criteria (measurable, in order of gating)
1. **Unit tests green** (pure CPU, no GPU): alignment walk, reward math, mask assembly with mocked `generate` — commands in §6.
2. **Micro-probe GO** (§4) on ~50 validation conversations × N=8 at T=0.8, run BEFORE any trainer step:
   - primary: **median within-group `reward_std` ≥ 0.05** and **`frac_collapsed_groups` (std &lt; 0.01) &lt; 0.50** — the exact GRPO admissibility bar from `docs/grpo_viability_investigation.md` §1 that every single-turn measurement failed (~0.77 collapsed at bqbxnqxw geometry);
   - anti-truncation: **mean model turns before stop ≥ 3.0** and mean coverage in (0.05, 0.95);
   - environment: probe asserts `trl.__version__ == "1.0.0"` and that the `env_mask` pathway exists in the *patched* trainer source (see §2.3).
3. **Mask audit passes** at rollout time: for every sample, `len(env_mask) == len(completion_ids)`; decoding the mask==1 positions reproduces exactly the concatenated model-turn texts (modulo special tokens); decoding mask==0 positions contains only injected gold user/tool text + turn scaffolding. Logged once at probe time and at trainer step 0.
4. **50-step diagnostic run** (inheriting all stabilizers from `configs/training/grpo_cat_a_diagnostic.yaml`): completes without kill-criteria breach (grad_norm &lt; 50 sustained, KL &lt; 10), `frac_reward_zero_std &lt; 0.30` (vs 0.70–1.0 historically), `train/reward_std` materially above 0.02, and the trajectory held-out composite non-degrading.
5. Done = criteria 1–4 met and the 1000-step config is a one-line change (`training_steps`).

### 2.2 The token-masking crux — RESOLVED

**Mechanism chosen: `rollout_func` + the `env_mask` extra field.** Verified line-by-line against `.venv/lib/python3.12/site-packages/trl/trainer/grpo_trainer.py` (TRL 1.0.0, 2549 lines):

- `rollout_func` contract (`:114-117`, `:1596-1613`): receives `(prompts, trainer)` with structured (conversational) prompts, must return `prompt_ids`, `completion_ids`, `logprobs`; **all other returned keys become per-completion extra fields**.
- **`env_mask` is a first-class supported channel**: at `:1646-1649`, when `self.tools` is falsy (ours is), `tool_mask = extra_fields.pop("env_mask", None)` — the comment reads *"Support custom env_mask from rollout_func (e.g., for environment feedback masking). Internally treated as tool_mask - marks model tokens (1) vs external tokens (0)."* This is exactly our need.
- Downstream, `tool_mask` is padded with 1 (`:1788-1794`), and:
  - **loss**: `_compute_loss:2265` → `mask = completion_mask * inputs["tool_mask"]`; every loss type multiplies by `mask` (`grpo`/`bnpo`/`dr_grpo`/`dapo` at `:2389-2407`), so injected tokens contribute **zero** policy-gradient and zero KL (KL enters `per_token_loss` before the mask product, `:2384-2385` then `:2389+`);
  - **attention**: `completion_mask` stays all-ones over injected tokens (`:1768`), so the model attends to gold user/tool context during logp computation — correct;
  - **metrics**: entropy/KL logging uses `masked_batch_mean` over the same product mask (`:2412-2425`); `num_items_in_batch` counts only mask==1 tokens (`:1653-1656`) — DAPO-style normalizers stay honest;
  - **liger path** honors it too (`:2148-2149`) — though we run the non-liger path.
- `logprobs: None` is safe: only consumed for vLLM importance-sampling correction (`:1885`, `:2381`) and OPSM (`:2303`), all gated off under `use_vllm=False` / default `off_policy_mask_threshold=None`; `None` skips padding at `:1778-1787`. Old-policy logps for the PPO ratio are **recomputed by the trainer itself** via a no-grad forward over our returned ids (`:1860-1882`, triggered because `gradient_accumulation_steps(1) % generate_every(4) != 0`) — valid for interleaved sequences since it operates on token ids.
- `env_mask` is **popped** from extra fields, so it does not leak into reward kwargs; our other extra fields (`trajectory`, `rollout_meta`) are merged per-completion into reward inputs at `:1949-1956` and forwarded to the reward at `:1156-1157` + `:1196-1198`. `GRPOConfig.remove_unused_columns` defaults to `False` (grpo_config.py:375-377), so the `ground_truth` dataset column also reaches the reward.

**Mechanisms considered and rejected:**
- `tools` + `_tool_call_loop` (`:1410-1587`): executes real Python tools and regenerates keyed on parsed structured `tool_calls`; it cannot inject **user** turns, and structured tool-call parsing depends on `parse_response`/`response_schema` the Gemma-4 tokenizer may not ship. Rejected.
- `environment_factory` (`:1712-1717`, `:446-474`): `reset()` only mutates the last prompt message before generation — no per-turn hook. Rejected.
- `GRPOTrainer` subclass overriding `_generate`: viable **fallback** if the Unsloth patch layer on the training box breaks the rollout_func path (§5 R2), but unnecessary otherwise.

**Two verified hazards with mandated mitigations:**
1. `mask_truncated_completions` (default `False`, grpo_config.py:744): if ever enabled, `:1796-1804` zeroes the *entire* completion whose last token isn't EOS/pad — i.e., precisely our divergence-truncated trajectories. Mitigation: wiring asserts it is False **and** the rollout guarantees every trajectory's final token is `tokenizer.eos_token_id` (force-appended with mask=0 when a turn hits its token budget; trainer's `self.eos_token_id = tokenizer.eos_token_id`, `:327`). This also keeps the `completions/clipped_ratio` metric meaningful.
2. `dr_grpo` normalizes by `max_completion_length` (`:2396-2397`). Raising it 512→4096 shrinks per-token loss ~8×, partially offset by ~6× more unmasked tokens per sequence. Keep `lr: 1.0e-6` for the first run and monitor `grad_norm`; raising to 2e-6 if updates are vanishing is **[IMPLEMENTER DECIDES]** after inspecting the 50-step diagnostic.

### 3a. Trajectory loader — one row per conversation

**File:** `src/llm_workflow_agents/training/grpo.py` (beside `_load_grpo_jsonl`, :161-302) with the pure pieces in `trajectory_rollout.py`.

@dataclass(frozen=True)
class GoldScript:

# grpo.py

**Algorithm** (mirrors `_load_grpo_jsonl`'s enrichment, drops its flattening):
1. Per JSONL line: re-enrich `messages[0]` via `build_enriched_system_prompt(raw, raw_msgs[0]["content"], force_rebuild=True)` (`data/system_prompt.py:60`), exactly as :211-224 does today.
2. `prompt_messages` = messages up to (excluding) the **first** assistant turn, `_slim_content`-stripped. Verified shapes: 495/501 sampled rows have first assistant at index 2 (`[system, user]` prompt), 6/501 at index 1 (opener rows → `[system]` prompt — precedent: the single-turn loader already emits system-terminated prompts, :248-251).
3. Segmentation: for gold assistant turn index `t` (0-based over assistant turns), `segments[t]` = the gold messages strictly between assistant turn `t` and assistant turn `t+1`, roles ∈ {tool, user} (verified adjacency: only `(assistant→tool→user)` and `(assistant→user)` occur). `segments[-1]` = trailing non-assistant messages (usually `[]` — conversations end on assistant).
4. `gold_transitions` from `ground_truth.state_sequence` (`[{from,to}]` → tuples). **Invariant to assert per row:** `len(gold_transitions) == n_gold_assistant_turns` (verified on data: every assistant turn carries exactly one `annotations.state_transition`; L4 sample: 20 == 20). Rows violating it: log + skip.
5. Emit dataset row `{"prompt": prompt_messages, "ground_truth": json.dumps({...})}` where the GT dict carries: `state_sequence` (full, ordered), `tool_calls` (flat), `terminal_state`, `terminal_reached`, `valid_transitions` — same JSON-string-column trick as :287-290 (pyarrow schema bypass).
6. Build `script_index[prompt_key(prompt_messages)] = GoldScript(...)`. **Collision check:** if a key repeats with a different `conversation_id`, drop the later row and log `trajectory_prompt_key_collision` (system prompts embed per-conversation graphs; collisions should be ~zero, but silence here would mis-script rollouts).
7. Overlong guard: rows whose templated prompt exceeds `max_prompt_length` tokens are dropped with a counter (TRL does not truncate prompts on the rollout_func path; a silent overrun would blow the 8192 budget).

Yields ~2,502 train / 290 validation rows (verified counts).

### 3b. In-process replay rollout

**File:** `src/llm_workflow_agents/training/trajectory_rollout.py`

```python
@dataclass(frozen=True)
class TrajectoryRolloutConfig:
    max_turns: int = 24                  # p90 gold length is 21
    per_turn_max_new_tokens: int = 256
    max_completion_tokens: int = 4096    # must equal GRPOConfig.max_completion_length
    stall_turn_limit: int = 2            # consecutive no-transition turns before stop
    temperature: float = 0.8
    top_p: float = 0.95
    do_sample: bool = True

@dataclass
class RolloutSample:
    prompt_ids: list[int]
    completion_ids: list[int]
    env_mask: list[int]                  # len == len(completion_ids); 1=model, 0=injected/forced
    turn_texts: list[str]                # decoded model turns, in order
    meta: dict                           # {"cursor": int, "stop_reason": str, "n_model_turns": int,
                                         #  "n_stall_turns": int, "gold_len": int, "conversation_id": str}

def run_replay_rollout(
    model, tokenizer,
    scripts: list[GoldScript],           # one per requested completion (duplicates for a group)
    cfg: TrajectoryRolloutConfig,
) -&gt; list[RolloutSample]

def make_replay_rollout_func(
    script_index: dict[str, GoldScript],
    cfg: TrajectoryRolloutConfig,
) -&gt; "RolloutFunc"                       # closure: (prompts, trainer) -&gt; dict
```

**`make_replay_rollout_func` returned callable** (the TRL-facing adapter):
1. `assert_trajectory_rollout_support()` on first call.
2. For each prompt in `prompts` (order-preserving — group advantage at `:1967` depends on it): `scripts.append(script_index[prompt_key(prompt)])`; a missing key is a hard error (never silently degrade to single-turn).
3. `samples = run_replay_rollout(trainer.model, trainer.processing_class, scripts, cfg)` — temperature/top_p read from `trainer.args`.
4. Return:
```python
{
  "prompt_ids":     [s.prompt_ids for s in samples],
  "completion_ids": [s.completion_ids for s in samples],
  "logprobs":       None,
  "env_mask":       [s.env_mask for s in samples],
  "trajectory":     [json.dumps(s.turn_texts) for s in samples],   # per-completion extra field
  "rollout_meta":   [json.dumps(s.meta) for s in samples],
}
```

**`run_replay_rollout` turn loop** (batched, active-set):
1. `prompt_ids` = `tokenizer.apply_chat_template(script.prompt_messages, add_generation_prompt=True, tokenize=True)`. Derive the assistant **turn-end token id(s)** once via a dummy render (`[user:"dummy", assistant:"dummy"]`, take trailing special token(s)); assert it includes `tokenizer.eos_token_id` — for Gemma chat tokenizers eos is `&lt;end_of_turn&gt;`; if they differ, generation must stop on the turn-end id and the force-append token must be `tokenizer.eos_token_id`-compatible (**[IMPLEMENTER DECIDES]** the exact reconciliation after inspecting the actual Gemma-4 SFT tokenizer; unit-test locks the invariant "trajectory ends on `tokenizer.eos_token_id`").
2. While any sample active: left-pad the batch of `prompt_ids + completion_ids`, call `model.generate(..., max_new_tokens=cfg.per_turn_max_new_tokens, do_sample=cfg.do_sample, temperature, top_p, eos_token_id=turn_end_ids)` under `torch.no_grad()` with `model.eval()` (restore mode after — mirror `_HeldOutEvalCallback._evaluate`, grpo.py:732-779). Per sequence, take generated ids up to and **including** the first turn-end token → append to `completion_ids` with `env_mask += [1]*len`. If no turn-end within budget: append the sampled ids, then **force-append the turn-end id with mask 0** (not sampled → not trained).
3. Decode the turn (`skip_special_tokens=True`) → `turn_texts.append`; parse transitions via `extract_state_annotations` (reward_utils.py:17-22 → `parse_state_transitions`, eval/state_accuracy.py:64-79).
4. **Alignment &amp; truncation predicate** — `classify_turn(pred_transitions, cursor, gold_transitions)` (pure, shared with the reward):
   - `effective = [t for t in pred_transitions if t[0] != t[1]]` (self-loops `(X,X)` are "stay" markers, legal per `transition_legality_score` semantics — neutral for alignment).
   - `effective == []` → `("stall", cursor)`.
   - `effective == gold_transitions[cursor : cursor+len(effective)]` (exact, order-sensitive, consecutive; ≥1 element) → `("advance", cursor+len(effective))`. Multi-step compression is allowed **only** as an exact consecutive gold run.
   - anything else → `("diverged", cursor)` — this *is* the "leaves the gold-reachable set" rule, keyed off gold `state_sequence` order, never a re-simulated transition function (the serialized graph has no trigger semantics — verified: `transitions[].condition` is an opaque label).
5. **Stop conditions**, checked after the turn (the turn's tokens always stay in the trajectory and are scored):
   - `diverged` → stop, `stop_reason="diverged"`;
   - `cursor == len(gold_transitions)` → stop, `"gold_complete"`;
   - consecutive stalls ≥ `stall_turn_limit` → stop, `"stall"`;
   - turn index `t+1 &gt;= min(len(segments), cfg.max_turns)` → stop, `"script_exhausted"` / `"turn_cap"`;
   - `len(completion_ids) + per_turn_max_new_tokens + len(next_segment_ids) + 8 &gt; cfg.max_completion_tokens` → stop, `"budget"`.
6. **Gold injection** (if continuing): compute segment ids with the **dummy-conversation diff** — the exact in-library technique of `_get_tool_suffix_ids` (grpo_trainer.py:1381-1408): render `[{user:"dummy"},{assistant:"dummy"}]` without generation prompt, render the same + `segments[t]` with `add_generation_prompt=True`, trim the prefix at its last eos, assert prefix property, take the suffix. Append suffix ids with `env_mask += [0]*len` (covers tool text, user text, and the next `&lt;start_of_turn&gt;model` header — mirroring `_tool_call_loop`'s mask update at :1551-1559). Tool results are injected **unconditionally** from gold, whether or not the model emitted the matching call (replay-scripted; same semantics as `_replay_conversation`'s "GT tool responses kept as-is", agent_benchmark.py:441-445). Cache segment-id computation per (script, t).
7. **Final-token guarantee:** if the loop ends with a mask-0 injected suffix pending or a non-eos tail, force-append `tokenizer.eos_token_id` with mask 0 (keeps `is_truncated` at :1674 False and immunizes against `mask_truncated_completions`).
8. **Mask invariants** (assert per sample, cheap): `len(env_mask)==len(completion_ids)`; `sum(env_mask) &gt; 0`; first token has mask 1.

Cost model: 32-sample generation batch × median ~8–13 rounds × ≤256 tokens ≈ 3–6× the single-turn ~31 s/step HF-rollout cost → ~2–4 min/generation cycle; 50-step diagnostic ≈ 2–3 H100-h. Trajectory v1 is **HF-generate only**: if the YAML requests `generation_backend: vllm` with trajectory mode, force HF with a warning (Gemma-4 can't use Unsloth vLLM anyway — Risk R9, grpo.py:526-543).

### 3c. Trajectory reward

**File:** `src/llm_workflow_agents/training/rewards/reward_business_logic.py` (append; register in `grpo.py:_REWARD_REGISTRY` as `"reward_business_logic_trajectory"`).

```python
def reward_business_logic_trajectory(
    prompts: list[Any],
    trajectories: list[list[str]],       # per-completion model turn texts
    metas: list[dict[str, Any]],         # rollout_meta (trusted: produced by our rollout, not the model)
    ground_truths: list[dict[str, Any]], # decoded ground_truth column
) -&gt; list[float]:
```

Per completion, with `gold = [(f,t)...]` from `gt["state_sequence"]`, `T = len(gold)` (≥1 by loader invariant):

1. **Recompute alignment** by replaying `classify_turn` over `extract_state_annotations(turn)` for each turn (same pure function the rollout used — single source of truth; warn-log if the recomputed cursor ≠ `meta["cursor"]`). Yields `cursor`, `n_stall_turns`, `pred_transitions_all` (flat, ordered), `pred_tools_all` (flat, ordered, via `extract_tool_calls` per turn).
2. Components:
   - `r_coverage = cursor / T` — **order-sensitive by construction** (cursor only advances on the exact next gold transition or an exact consecutive gold run) and **capped at 1.0 structurally** (cursor ≤ T).
   - `r_tool = graded_tool_call_f1(pred_tools_all, _strip_placeholder_args(gt["tool_calls"]))` (reward_utils.py:50-60 → `compute_argument_graded_f1`, eval/tool_call_f1.py:258; placeholder-stub handling :116-131). Gold side = the **whole conversation's** tool calls (locked decision 3): recall scales with progress, precision punishes spam.
   - `r_terminal = 1.0` iff `meta["stop_reason"] == "gold_complete"` **and** `pred_transitions_all` is non-empty **and** `pred_transitions_all[-1][1] == gt["terminal_state"]`; component **active only when** `gt["terminal_reached"]` (else dropped + weights renormalized — existing pattern, reward_business_logic.py:181-184).
   - `r_legality = transition_legality_score(pred_transitions_all, gt["valid_transitions"]) * stall_factor`, where `stall_factor = 1 - n_stall_turns / max(n_model_turns, 1)`; component active only when `valid_transitions` non-empty (:173-176 pattern).
3. `score = Σ wᵢsᵢ / Σ wᵢ` over active components with `W = (0.40, 0.40, 0.10, 0.10)`; clip to [0,1].

**Anti-hacking guards** (locked decision 3, made concrete): order-sensitivity + coverage cap via the cursor; stall farming punished twice (stall_factor + the rollout's consecutive-stall stop); transition spam ends the rollout (divergence) *and* depresses legality; tool spam depresses F1 precision; and the independent strict held-out composite (§3d) + `_is_reward_hacking` auto-stop (grpo.py:451-468) remains the backstop.

**Adapter** — `grpo.py::_make_trajectory_reward_adapter(reward_fn)`: same keyword contract as `_make_reward_adapter` (:334-340) but reads `kwargs["trajectory"]` and `kwargs["rollout_meta"]` (per-completion extra fields, JSON-decoded) **instead of collapsing `completions` to the last message** (:363-370 — that collapse would hand the reward TRL's `batch_decode` of the whole interleaved stream, :1629-1630, which must not be scored). Missing `trajectory` key → hard error. Keep the `_LATEST_INSTRUMENTATION` unique-completions stash (:376-387), keyed on prompt, using `"\n".join(turn_texts)` as the completion identity.

## 7. What would falsify this design (abandon → RFT/SFT-only)

1. **Variance doesn't materialize:** probe `median_reward_std &lt; 0.02` while `mean_model_turns ≥ 3` — trajectories are long but groups still land on one rung (e.g., the SFT policy walks the gold path near-deterministically at T=0.8, coverage ≈ 1.0 everywhere). The premise "aggregation manufactures variance" is wrong for this policy/data; the task is trajectory-saturated → ship SFT-only or RFT.
2. **Over-truncation floor is structural:** `mean_model_turns &lt; 2` even after stall-limit 3 and one-legal-edge tolerance — the policy exits the gold-reachable set immediately, so on-policy multi-turn conditioning has no support. Free-run + gold-script replay (locked decision 2) cannot work; fall back to RFT on gold-context turns.
3. **Stable but flat, again:** 50-step diagnostic with healthy `reward_std` (&gt; 0.05) and sane optimization (grad_norm, KL nominal) yet reward mean and held-out composite immobile after the LR check (R6) — variance was necessary but not sufficient; step economics at lr ~1e-6 can't move a 26B policy on this signal. Kill per the same step-economics argument that killed single-turn (`docs/grpo_viability_investigation.md` §1).
4. **Reward-quality divergence:** training reward climbs while the strict trajectory held-out composite falls across ≥2 consecutive evals (auto-stop fires) and transcript audit confirms degenerate coverage-farming — the blend is hackable in practice despite the guards; retire the online optimizer for good.
5. **Mechanism unsound on the real stack:** the chat-template prefix-diff assertion (`_get_tool_suffix_ids`-style) cannot be satisfied for the Gemma-4 tokenizer (template not prefix-stable), or the training box's Unsloth-patched trainer loses `env_mask` with no subclass workaround — the masking correctness guarantee is gone; do not train with an unverifiable mask.

```python
        # Active components: (weight, score) pairs. task_completion and
        # transition_legality are conditionally active.
        components: list[tuple[float, float]] = [
            (W_STATE_TRANSITION, r_state),
            (W_TOOL_CALL_F1, r_tool),
        ]

        valid_transitions = gt.get("valid_transitions", [])
        if valid_transitions:
            r_legality = transition_legality_score(pred_states, valid_transitions)
            components.append((W_TRANSITION_LEGALITY, r_legality))

        # terminal_reached=False means the conversation never reached a terminal
        # state in the ground truth (e.g. adversarial L4/L5 timeouts, or this
        # is a non-final per-turn row). Skip the completion sub-reward.
        if gt.get("terminal_reached", True):
            terminal = gt.get("terminal_state", "")
            r_completion = 1.0 if reached_terminal(completion, terminal) else 0.0
            components.append((W_TASK_COMPLETION, r_completion))

        total_weight = sum(w for w, _ in components)
        score = (
            sum(w * s for w, s in components) / total_weight
            if total_weight &gt; 0
            else 0.0
        )
        rewards.append(max(0.0, min(1.0, score)))
```
