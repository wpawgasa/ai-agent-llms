# Fine-Tuning Recipes

This document describes the two-stage fine-tuning pipeline for the 3 Phase 1 category winners.
Training is driven by `training/sft.py` and `training/grpo.py` using Unsloth on a single H100 SXM 80GB.

---

## Overview: Two Stages

```
Phase 1 winner (pre-trained)
        │
        ▼
  [Stage 1: SFT]  ── 03-training.md § sft.py
        │  format/domain alignment, ConstantLengthDataset packing
        ▼
  SFT checkpoint
        │
        ▼
  [Stage 2: GRPO]  ── 03-training.md § grpo.py
        │  task-metric RL, vLLM generation backend, DAPO normalization
        ▼
  GRPO checkpoint
        │
        ▼
  merge_adapter.py  →  merged BF16 or FP8 weights  →  Phase 3 quantization
```

---

## Pre-Flight: Pilot Check (Risk R3)

Before committing to a full SFT run, validate the Phase 1 winner responds to fine-tuning.

```bash
source .venv/bin/activate && python -m training.pilot_check \
  --models <rank1_model_id> <rank2_model_id> \
  --task_data data/output/task_a \
  --pilot_steps 100
```

`pilot_check.py` runs 100-step SFT on the top-2 candidates. If rank-1 shows validation loss
divergence or metric regression, it auto-selects rank-2 before proceeding.

## System-Prompt Enrichment (train/eval alignment)

Task A training samples store a fully enriched `messages[0]` — the system prompt already contains the workflow script, structured reference (initial state, terminal states, tool names), and the 7-rule format guide. This is produced at generation time by `data/system_prompt.py::build_enriched_system_prompt` and applied to every sample regardless of whether it was generated by a teacher model or the placeholder generator.

This means training and benchmark eval see **the same prompt shape**: the model learns to ground its responses on the structured workflow context it will receive at inference. No special handling is required in the SFT trainer — the enriched content is already in the stored JSONL.

The benchmark harness (`eval/agent_benchmark.py`) keeps its own call to `build_enriched_system_prompt` as a backward-compat safety net for legacy bare-prompt files. On new data the call is a no-op (the function is idempotent).

Approximately **30% of Task A SFT samples** use a teacher-authored rich natural-language system prompt in `messages[0]` — a structured voicebot-style document with a persona line, a `## GOAL` section, and per-state `### [state_name]` blocks containing dialogue hints and intent-based branches (see `generate_workflow_dataset`'s `rich_prompt_rate` parameter). The remaining 70% use the bare one-line role. In both cases the trailing structured reference is identical (workflow script + tool list + format rules). This ensures the model is robust to both prompt shapes at inference, since production deployments may supply either. No SFT trainer change is required — the diversity is already encoded in the stored JSONL.

## Stage 1: Supervised Fine-Tuning

### Entry point

```bash
source .venv/bin/activate && python -m training.sft \
  --config configs/training/sft_cat_a.yaml   # or sft_cat_b.yaml / sft_cat_c.yaml
```

### Pipeline steps

1. **Model load** — `FastLanguageModel.from_pretrained()`
   - MoE models (Qwen3.5-35B, Qwen3.6-35B, GLM-4.7-Flash, Nemotron-30B): `load_in_4bit=True` (QLoRA, ~17.5 GB active)
   - Dense models ≤8 GB active params: BF16

2. **LoRA setup** — `FastLanguageModel.get_peft_model()`
   - Target modules resolved from `training/lora_targets.py` (per-architecture registry)
   - Default rank 64, alpha 64, dropout 0
   - MoE: `freeze_router=True` (freeze `mlp.gate` weights during SFT)
   - GLM-4.7-Flash: auto-fallback rank 64→32 if VRAM OOM (Risk R7)

3. **Data packing** — `ConstantLengthDataset` with model-specific chat template
   - Templates from `data/chat_template_converter.py` (6 formats: qwen, qwen35, gemma, mistral_instruct_v3, nemotron, glm_chatml)
   - Split: 85% train / 10% val / 5% test

4. **Training loop** — SFTTrainer
   - Gradient checkpointing: `use_gradient_checkpointing="unsloth"`
   - Checkpoint every 500 steps; select best by validation loss

### Config reference (`configs/training/sft_cat_a.yaml`)

| Key | Value |
|-----|-------|
| `stage` | `sft` |
| `framework` | `unsloth` |
| `lora.rank` | 64 |
| `lora.target_modules` | `auto` (resolved from registry) |
| `training.precision` | `qlora_4bit` (MoE) or `bf16` (dense) |
| `training.lr_scheduler` | cosine |
| `training.warmup_ratio` | 0.03 |
| `training.num_epochs` | 3 |
| `training.max_seq_length` | 8192 |
| `training.packing` | true |
| `data.source` | `data/output/task_a` |
| `logging.wandb_project` | `llm-workflow-agents-sft` |

### Per-category data sources

| Category | Data dir | Task |
|----------|----------|------|
| Cat A | `data/output/task_a` | Multi-turn workflow conversations (L1–L5, 1000 samples) |
| Cat B | `data/output/task_b` | Tool-call data (external + 15K synthetic, 15% negatives) |
| Cat C | `data/output/task_c` | (prompt, graph) pairs (5000 samples) |

## Stage 2: GRPO Reinforcement Learning

### Entry point

```bash
source .venv/bin/activate && python -m training.grpo \
  --config configs/training/grpo_cat_a.yaml   # or grpo_cat_b.yaml / grpo_cat_c.yaml
```

### Pipeline steps

1. **Load SFT checkpoint** — `FastLanguageModel.from_pretrained(sft_checkpoint_path)`
2. **Configure GRPOTrainer** with task-specific reward function
3. **Train** 500–1000 steps with monitoring
4. **Auto-stop** if held-out metric drops while reward rises (hacking detector)

### Reward functions

**Cat A — `training/rewards/reward_business_logic.py`**

| Component | Weight | What it measures |
|-----------|--------|-----------------|
| `state_transition_correctness` | 0.30 | `[STATE: X → Y]` annotations match ground truth |
| `tool_call_f1` | 0.30 | AST sub-tree match of `<tool_call>` JSON vs gold |
| `chain_propagation_accuracy` | 0.20 | Correct multi-hop tool dependencies |
| `format_compliance` | 0.10 | Well-formed state + tool annotations |
| `task_completion` | 0.10 | Reached correct terminal state |

**Cat B — `training/rewards/reward_subagent.py`**

| Component | Weight |
|-----------|--------|
| `tool_call_f1` | 0.40 |
| `slot_extraction_acc` | 0.30 |
| `state_sequence_match` | 0.20 |
| `format_compliance` | 0.10 |

**Cat C — `training/rewards/reward_graph_extraction.py`**

| Component | Weight | Notes |
|-----------|--------|-------|
| `json_validity_bonus` | 0.10 | Early exit → 0.0 if invalid JSON |
| `node_f1` | 0.35 | |
| `edge_f1` | 0.35 | |
| `structural_validity` | 0.10 | Valid initial state, reachable terminals, no orphans |
| `1 - normalized_GED` | 0.10 | Graph edit distance vs gold |

All rewards return `list[float]` in `[0.0, 1.0]`. Shared helpers in `training/reward_utils.py`.

### Key GRPO hyperparameters (`configs/training/grpo_cat_a.yaml`)

| Key | Value | Why |
|-----|-------|-----|
| `grpo.normalization` | `DAPO` | Removes length bias from reward signal |
| `grpo.num_generations` | 4 | Samples per prompt for group relative scoring |
| `grpo.beta` | 0.04 | KL penalty; keeps policy near SFT checkpoint |
| `grpo.epsilon` | 0.2 | PPO clip ratio |
| `grpo.learning_rate` | 5e-6 | Lower than SFT to avoid catastrophic forgetting |
| `grpo.training_steps` | 1000 | |
| `grpo.generation_backend` | `vllm` | 11× faster RL inference vs HF generate |
| `grpo.fp8_rl` | true | H100: 1.4× faster, 60% less VRAM during RL rollouts |

### Monitoring & hacking guard

| Signal | Frequency | Action |
|--------|-----------|--------|
| Reward curve | every step | W&B log |
| Held-out eval | every 50 steps | Compare task metric vs reward |
| KL divergence | every step | W&B log |
| Auto-stop | on detection | Halt if held-out ↓ while reward ↑ |

## Post-Training: Adapter Merge

```bash
source .venv/bin/activate && python -m training.merge_adapter \
  --base_model <hf_model_id> \
  --adapter_path checkpoints/grpo_cat_a/best \
  --output_path checkpoints/merged_cat_a \
  --quantize_merged fp8          # omit to keep BF16
```

`merge_adapter.py` calls `model.merge_and_unload()` then optionally exports to FP8 for Phase 3 quantization benchmarking.

## LoRA Target Module Registry

`training/lora_targets.py` maps model family → target modules:

| Model family | LoRA targets | Notes |
|---|---|---|
| Qwen2.5-3B, Qwen3-32B, Gemma-2B, Gemma3-4B, Mistral-24B | q/k/v/o + gate/up/down_proj | Standard dense |
| Qwen3.5-4B, Qwen3.5-35B-A3B, Qwen3.6-35B-A3B | + DeltaNet: in_proj_qkv/z/b/a, out_proj; freeze mlp.gate | MoE, QLoRA 4-bit for 35B |
| GLM-4.7-Flash | MLA: q_a/q_b/kv_a_proj_with_mqa/kv_b/o_proj + shared_experts; freeze mlp.gate | Auto-fallback rank 64→32 |
| Nemotron-30B | Unsloth auto-detect Mamba layers; freeze mlp.gate | vLLM compat uncertain (Risk R6) |
| Gemma4-26B-A4B, Gemma4-31B, Gemma4-E4B, Gemma4-E2B | Same pattern as Gemma3; freeze mlp.gate | Added in Phase 1 expansion |

## Reproducibility

- SFT: seed passed via config; HF `set_seed()` called before trainer init
- GRPO: seed-deterministic rollout via vLLM `--seed`
- All checkpoints include the resolved config YAML for exact replay
