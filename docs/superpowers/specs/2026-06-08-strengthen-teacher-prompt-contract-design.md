# Strengthen Teacher Prompt to Prevent Placeholder Fallback — Design

**Date:** 2026-06-08
**Status:** Approved (brainstorming) → ready for implementation plan

## Context / Problem

Task A conversations can be generated by a teacher model (e.g. `gemini-3-5-flash`). After
generation, `generate_workflow_dataset` runs an inline **repair loop**: if a teacher
conversation is *incoherent* it retries up to `max_repair_retries`, then falls back to the
deterministic **placeholder generator** so the sample still respects the curated workflow.

Reviewing a real teacher run surfaced a placeholder fallback in the wild: sample `L1_010`
(outbound banking/`loan_offer`) had the bare fallback system prompt and placeholder text
(`"Handling Banking … in state GREETING"`, `"[Turn 1] Oh, hi…"`) — its teacher output was
incoherent and got replaced. Placeholder fallbacks are lower-quality than genuine teacher
dialogue, so we want to **reduce how often the teacher trips the repair loop**.

A conversation is judged incoherent by exactly two checks:

1. **Transition legality** — `_find_transition_violations` (`generate_workflows.py`) flags any
   assistant `[STATE: X → Y]` annotation with `X ≠ Y` that is **not** a legal edge in the
   workflow graph. In-state (`X → X`) annotations are always allowed.
2. **Tool placement** — `find_tool_placement_violations` (`_workflow_script.py`) flags any tool
   called from a state whose curated tool list does not include it, or a tool not present in the
   sample's tool schemas at all.

The current teacher prompt hands the model the workflow graph as JSON plus generic prose rules
("never make invalid transitions"). It does **not** enumerate the legal transitions or the
per-state tool permissions in an imperative, hard-to-miss form, so the model must derive
legality from JSON — and sometimes gets it wrong. Worse, `FORMAT_RULES` rule 3 tells the model
that per-state tool listings are "UNRELIABLE" and to trust the schema, which actively works
against satisfying the tool-placement check.

## Goal

Make the teacher prompt state the repair-loop constraints **explicitly and authoritatively**, so
a teacher that follows the prompt cannot trip either check. Reduce placeholder fallbacks without
touching the repair loop, retries, or fallback mechanism.

## Decisions (from brainstorming)

- **Scope: prompt-only.** No changes to the repair loop, retry count, fallback mechanism, or
  `FORMAT_RULES`. Changes are confined to teacher-generation code paths.
- **Contract covers both violation types** — legal transitions *and* per-state tool placement —
  with rule 3's tension reconciled (schema = which tools exist + their arguments; contract =
  where tools may be called + which transitions are legal; workflow script = flow narrative only).
- **Verification: structural test** that pins the rendered contract to the repair-loop checks
  (no live API run required in this session).

## Design

### 1. Contract helpers (`generate_workflows.py`)

Two pure functions, deriving from the **same** data the repair loop checks against:

```python
def _workflow_contract(workflow: WorkflowGraph) -> tuple[set[tuple[str, str]], dict[str, list[str]]]:
    """Return (legal_directed_edges, tools_by_state) keyed by state NAME.

    legal_directed_edges: {(from_name, to_name) for each transition where from_name != to_name},
        using the same id->name map the repair loop uses
        (id_to_name = {s.id: s.name for s in workflow.states}).
    tools_by_state: {s.name: sorted(s.tools) for s in workflow.states}.
    """

def _render_workflow_contract(workflow: WorkflowGraph) -> str:
    """Render the (edges, tools_by_state) pair into the prompt CONTRACT block (below)."""
```

`_workflow_contract` must mirror the repair-loop derivations exactly:
- edges ↔ the `X != Y` subset of `valid_edge_pairs` built in `generate_workflow_dataset`
  (`{(id_to_name[t.from_state], id_to_name[t.to_state]) for t in workflow.transitions}`).
- tools_by_state ↔ `allowed = {s.name: set(s.tools) for s in workflow.states}` used by
  `find_tool_placement_violations`.

### 2. Rendered CONTRACT block (inserted into `_build_teacher_prompt`)

Placed near the **top** of the teacher user prompt — after the domain/behavior/outbound lines,
**before** the JSON graph dump — so it is the most prominent legality statement:

```
WORKFLOW CONTRACT — these are hard constraints. Output that violates them is rejected and regenerated.

ALLOWED TRANSITIONS (only these state changes are legal; staying in the same state is always allowed):
  GREETING → VERIFY_IDENTITY
  VERIFY_IDENTITY → TERMINAL
TOOL PERMISSIONS PER STATE (a tool may ONLY be called from a state that lists it):
  GREETING: (text only — no tools)
  VERIFY_IDENTITY: verify_identity
  TERMINAL: (text only — no tools)

Authority: the tool SCHEMAS define which tools exist and what arguments they take; this CONTRACT
defines where each tool may be called and which transitions are legal; the workflow script is only
a flow hint. When they disagree, the schema wins for arguments and the CONTRACT wins for placement
and transitions.
```

Edges and per-state tool lines are rendered in a stable, sorted order for determinism.

### 3. Reinforcing rules (`_TEACHER_SYSTEM_PROMPT`)

Add two rules to the teacher **system** prompt (used only by `_generate_teacher_conversation`, so
no training/eval impact):

- *"Every `[STATE: X → Y]` you emit with X ≠ Y MUST appear in the WORKFLOW CONTRACT's ALLOWED
  TRANSITIONS list. Never invent a transition; if unsure how to proceed, stay in the current
  state (`[STATE: X → X]`)."*
- *"Only call a tool from a state that lists it under TOOL PERMISSIONS PER STATE. Never call a
  tool in a state marked 'text only', and never call a tool that is absent from the tool
  schemas."*

`FORMAT_RULES` is **not** modified; the rule-3 reconciliation lives inside the CONTRACT block's
"Authority" line, scoped to teacher generation only.

### 4. Structural test (`tests/unit/test_teacher_prompt_contract.py`)

For a representative sweep of `(domain × level)` subgraphs (seeded), build the teacher prompt and
independently recompute what the repair loop accepts, then assert:

- `_workflow_contract(workflow)` edge set **equals** the repair loop's legal directed edges
  (the `X != Y` subset of `valid_edge_pairs`) — no missing, **no extra**;
- per state, `_workflow_contract(workflow)` tools **equal** `allowed_tools_by_state[state]`;
- the rendered prompt text (`_build_teacher_prompt(...)`) **contains** every contract line
  (each `from → to` edge and each `state: tools` line).

This guarantees the prompt's contract is a faithful, complete representation of the checks: a
teacher obeying the prompt literally cannot trip the repair loop. The existing unit suite must
stay green (no regression in `TestTeacherPromptOutbound` etc.).

## Files Touched

| File | Change |
|------|--------|
| `src/llm_workflow_agents/data/generate_workflows.py` | `_workflow_contract`, `_render_workflow_contract`; insert CONTRACT block into `_build_teacher_prompt`; add 2 rules to `_TEACHER_SYSTEM_PROMPT` |
| `tests/unit/test_teacher_prompt_contract.py` | New structural / no-drift test |

## Out of Scope

- The repair loop, `max_repair_retries`, and the placeholder fallback mechanism (unchanged).
- `FORMAT_RULES` and the training/eval enrichment path (`build_enriched_system_prompt`) — unchanged.
- Live before/after fallback-rate measurement (needs a teacher API key; can be run separately).

## Verification

1. `python -m pytest tests/unit/test_teacher_prompt_contract.py -q` → structural/no-drift test passes.
2. `python -m pytest tests/unit -q` → full suite green (no regression).
3. (Optional, separate) A live `generate_benchmark_data_teacher.sh` run with an API key, comparing
   the `repair_fallbacks` stat before/after on the same seed.
