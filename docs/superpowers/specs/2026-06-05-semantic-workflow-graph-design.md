# Semantic Workflow Graph Generation — Design Spec

**Date:** 2026-06-05
**Status:** Approved design, pending implementation plan
**Area:** Task A data generation (`src/llm_workflow_agents/data/`)

## Problem

Task A workflow graphs are generated by sampling a random number of states, naming
them by cycling through a domain's flat `state_templates`, and wiring transitions
**by positional index** independent of state names. Transition conditions are
mechanical labels (`branch_S3_to_S5`). This produces semantically-incorrect graphs,
and — critically — the errors reach the **ground truth** used for eval and GRPO reward,
not just the prompt hints.

Defects confirmed in this codebase (see investigation in session 2026-06-04/05):

1. **Name-cycling duplication.** A domain has ~8–12 `state_templates`, but L4/L5 demand
   13–30 states. The generator cycles names (`(i-1) % len(middle)`), producing up to 8
   duplicate state names at L5 (`generate_workflows.py:795-798`).
2. **Name-blind transitions.** Transitions are wired `S_i → S_{i+1}` by index plus random
   forward skip-edges (`:814-836`), ignoring state semantics entirely. At L5 this makes even
   the priority-0 happy-path backbone regressive (`RESOLVE → VERIFY_IDENTITY`), and produces
   self-loops and gate-skipping branches (e.g. `GREETING → LOOKUP_ACCOUNT`, skipping identity
   verification).
3. **Meaningless conditions.** Branch conditions are `branch_S{src}_to_S{dst}`; rendered in
   the script they reduce to a generic "alternative condition met" fallback.
4. **GT corruption.** The placeholder generator walks `states[]` by list index — *not* the
   transitions — so the duplicate-name regressions land verbatim in
   `ground_truth.state_sequence` (proven: `RESOLVE → VERIFY_IDENTITY` appears twice in L5 GT).
   The teacher path is handed the malformed graph as authoritative and told to follow it; the
   repair loop only checks tool placement, never transition validity.
5. **Never-reaches-terminal.** The placeholder turn cap (`min(len(states)*2, 20)`) means L5
   conversations (30 states) stop mid-flow with `terminal_state = ""` — every L5 placeholder
   sample has a degenerate task-completion target.
6. **Upsell not modeled.** Upsell/promo is prompt-flavor only (an intent tag + one teacher-prompt
   line); the state machine and GT are identical for service vs upsell conversations.

## Goals

- Generated workflow graphs are **semantically correct by construction**: no gate-skips, no
  duplicate state names, no graph-edge self-loops, no regressive happy-path edges.
- Transition conditions are **meaningful** (human-readable labels) and **consistent** with the
  path the conversation actually walks, so GT is correct.
- Conversations always reach a terminal state.
- Upsell/promo is a semantically distinct, correct sub-path in the graph + GT.

## Non-Goals

- Regenerating the published benchmark/SFT corpora is a **downstream follow-up run**, not part of
  this code change.
- No changes to the eval metrics, reward functions, or training loop beyond what the new GT shape
  requires (the GT *shape* is unchanged; only its *correctness* improves).
- No change to the `workflow_graph` / `ground_truth` JSONL output schema (verified compatible).

## Resolved Design Decisions

| Decision | Choice |
|----------|--------|
| Complexity ↔ curated graph | **Subgraph of one canonical per-domain graph.** Complexity = subgraph richness (path length + branches + loops), not a random raw state count. |
| Transition conditions | **Typed + authored label.** Each edge carries a readable `label` plus a `trigger` type; the walker picks the edge whose trigger matches the simulated turn outcome. |
| Scope / migration | **Full replacement.** Rewrite the `DomainSpec` schema, re-author all 18 domains, rewrite generator/GT/script, delete the old index-walk path. No dual model. |
| Domain expansion | **Hybrid.** Expand ~5 rich domains to ~16–20 states; rescale L4/L5; constrain L4/L5 sampling to capable domains. |
| Upsell modeling | **Explicit optional graph arc** that rejoins the spine toward terminal, for the 15 domains with upsell intents. |

## Section 1 — New `DomainSpec` schema

Collapse the three parallel flat dicts (`state_templates`, `state_tools`, `state_instructions`)
into one `states` list, and add an explicit `edges` list — the canonical, semantically-valid
transition graph.

```python
@dataclass
class StateNode:
    name: str
    instruction: str
    tools: tuple[str, ...] = ()          # tools legitimately used in this state
    kind: str = "working"                # "initial" | "working" | "terminal"

@dataclass
class Edge:
    src: str
    dst: str
    label: str                           # human-readable: "identity already on file"
    trigger: str                         # always | tool_success | tool_error
                                         #   | intent_match | slot_present | user_declines
    optional: bool = False               # spine (False) vs branch/loop (True)
    priority: int = 0
    intent_category: str | None = None   # set on upsell-arc edges: "upsell_promo"

@dataclass
class DomainSpec:
    name: str
    category: str
    tools: tuple[dict, ...]              # OpenAI tool schemas — unchanged
    intents: tuple[str, ...]
    entity_slots: tuple[str, ...]
    states: tuple[StateNode, ...]        # replaces the 3 flat dicts
    edges: tuple[Edge, ...]              # NEW canonical transitions + conditions
    initial: str
    terminals: tuple[str, ...]
    intent_categories: dict[str, str] = field(default_factory=dict)
                                         # intent -> "service" | "upsell_promo"
                                         # (folds in INTENT_CATEGORY_TAXONOMY per-domain)
```

### `validate_domain()` invariants (enforced at import)

- Every `edge.src` / `edge.dst` references a known state name.
- `initial` has `kind="initial"`; every `terminals` entry has `kind="terminal"`.
- Every state is reachable from `initial`, and at least one terminal is reachable from every
  state (BFS over edges).
- Each non-terminal state has ≥1 outgoing edge and **exactly one spine successor**
  (`optional=False`, trigger `always` or `tool_success`) — guarantees a happy path exists.
- `trigger ∈ {always, tool_success, tool_error, intent_match, slot_present, user_declines}`.
- `tool_success` / `tool_error` triggers only on states whose `tools` is non-empty.
- **No graph-edge self-loops** (`src == dst` forbidden). Retries/loops are modeled as edges back
  to an *earlier distinct* state (e.g. `AUTH → VERIFY`).
- Upsell-arc edges (`intent_category="upsell_promo"`) must rejoin the spine toward a terminal
  (no upsell dead-ends; "secondary arc, not a hijack").

These invariants make the gate-skip, self-loop, duplicate-name, and regression defects
**structurally impossible** — the generator can only emit edges that were authored as valid.

> **Self-loop clarification (recipe compliance):** graph-*edge* self-loops are forbidden, but the
> conversation walker still emits turn-level `[STATE: X → X]` annotations when a state invokes a
> tool / handles a follow-up. Those message-level self-loops remain legitimate and appear in GT,
> matching `data_generation_recipes.md` lines 260/288.

## Section 2 — Complexity = subgraph selection

`ComplexitySpec` (`config/schema.py`) stops sampling a random `num_states` range. New fields drive
selection over the canonical graph:

```
target_path_len    # states along the main spine to include (capped at domain spine length)
num_branches       # optional branch edges to add
num_loops          # retry/escalation loop edges to add (0 for low levels)
include_recovery   # include tool_error recovery arcs
```

`num_states`, `num_tools`, `chain_depth` remain in the output as **derived counters** computed
from the selected subgraph.

### Re-scaled complexity levels

| Level | States | Branches | Loops | Recovery | Eligible domains |
|-------|--------|----------|-------|----------|------------------|
| L1 | 3–4 | 0 | 0 | no | all 18 |
| L2 | 5–7 | 1 | 0 | no | all 18 |
| L3 | 8–12 | 2–3 | 0–1 | optional | all 18 |
| L4 | 12–16 | 3–5 | 1 | yes | domains with ≥12 canonical states |
| L5 | 16–20 | all | 1–2 | yes | the 5 expanded rich domains (≥16 states) |

**Eligibility is derived, not hardcoded:** a domain serves level L iff its canonical state count
≥ L's minimum `target_path_len`. After expansion this yields: L1–L3 = all 18 domains;
L4 = domains with ≥12 states (`banking`, `travel`, plus the 5 expanded — note `banking`/`travel`
are in the expanded set); L5 = the 5 expanded rich domains (≥16 states). The `_select_domain`
filter computes this from the registry at runtime, so adding/expanding a domain automatically
updates its eligible levels.

The doc's original L4 (13–20) / L5 (21–30) ranges are retired: 21–30 distinct states is
unrealistic for a single customer-service workflow and was only ever reached via the cycling bug.

## Section 3 — Generator / GT / script rewrite

Two new pure functions replace the random index walk:

- **`select_subgraph(domain, complexity, rng) → WorkflowGraph`** — connected valid subgraph:
  walk the spine to `target_path_len`; add `num_branches` optional edges (+ their states);
  add `num_loops` back-edges to earlier distinct states; add `tool_error` recovery arcs if
  `include_recovery`; add the **upsell arc** when the sample's `intent_category == upsell_promo`
  and the domain has one. Emits the existing `workflow_graph` dict shape, with `condition`
  carrying the authored `label` and `priority` marking spine(0)/branch(1).

- **`walk_path(subgraph, behavior, intent_category, rng) → list[edge]`** — traverse from
  `initial`, at each state choosing the outgoing edge whose `trigger` matches the simulated
  turn outcome:
  - `tool_success` / `tool_error` — from the existing ~20% error draw at tool states.
  - `intent_match` — from the sample intent (drives the upsell arc).
  - `user_declines` / `slot_present` — from the behavior preset (adversarial declines;
    cooperative supplies the slot).
  - `always` — unconditional spine edge.
  Guaranteed to reach a terminal (selection guarantees reachability; walk prefers the spine when
  no trigger fires).

Consumers:

- **Placeholder generator** walks `walk_path` output instead of `current_state_idx + 1`, emitting
  `[STATE: from→to]` for real transitions and `[STATE: X→X]` for in-state tool turns. The turn cap
  is raised / made path-length aware so a terminal is always reached.
- **`_extract_ground_truth`** is mechanically unchanged (reads annotations) — but its input is now
  a valid path, so GT is correct by construction.
- **Teacher path** (`_build_teacher_prompt`) renders the same valid subgraph + typed conditions.
  The repair loop gains a **transition-validity check** (extend `find_tool_placement_violations`)
  so teacher-emitted `[STATE]` lines off the authored edge set are repaired/retried.
- **Script rendering** (`_graph_to_script` / `build_workflow_script`) emits the authored `label`
  (`- If identity already on file: go to [LOOKUP]`) instead of `branch_S3_to_S5`.

## Section 4 — Domain expansion + level rescaling

- `ComplexitySpec` fields shift from a sampled range to the selection drivers above.
- Add a **domain → max-level capability** map (derived from each domain's state count).
  `_select_domain` becomes **level-aware**: a domain serves level L iff its canonical state count
  ≥ L's minimum `target_path_len` (computed from the registry at runtime). In practice L4 = domains
  with ≥12 states, L5 = the 5 expanded rich domains. This is a deliberate, documented deviation
  from the doc's "domains fully decoupled from complexity" claim.
- **Expand 5 domains** — `banking`, `insurance`, `healthcare`, `travel`, `telecom` — to ~16–20
  real states with genuine sub-flows (document collection, assessment, escalation, recovery,
  appeal) + upsell arc. **No filler states.**
- Re-author the other 13 domains into the edge-graph schema at their current ~8–12 states
  (now with explicit edges + conditions).

## Section 5 — Validation & testing

- `validate_domain()` runs at import for all 18 (a test iterates `DOMAIN_REGISTRY` and asserts
  each passes).
- Property tests per level × eligible domain assert:
  - every GT transition is a valid edge in the emitted subgraph;
  - no unauthorized semantic regression (no edge to an earlier spine state except authored loops);
  - a terminal is always reached (`terminal_state != ""`);
  - **no duplicate state names** in any emitted graph;
  - upsell samples traverse the upsell arc; service samples do not.
- The investigation "proof" scripts (duplicate-name detector, regression detector,
  empty-terminal check) become regression tests.
- Update `test_data_generation.py` (new `num_states` semantics), `conftest.py` (`ComplexitySpec`
  fields), `test_webui.py` (`num_states`).
